> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Consolidate 3 copies of rules into 1 source of truth, remove dead files, and update all documentation to reflect the v0.8.0 plugin-based architecture.

**Architecture:** Rules live only in `rules/` at repo root (the plugin reads them from there). The `src/taleemabad_data_mcp/rules/` copy is removed вЂ” the Python package reads rules from `rules/` relative to repo root. The `.claude/rules/` dev copy is a symlink (Unix) or generated by the bump command. The bump command syncs `rules/` в†’ user's `~/.claude/rules/taleemabad/` on setup.

**Tech Stack:** Python 3.11, Claude Code plugin system, Git

### Task 3: Update cli.py to use repo-root rules/

**Files:**
- Modify: `src/taleemabad_data_mcp/cli.py`

The `_bundled_rules_dir()` function currently returns `Path(__file__).parent / "rules"` (the src copy we just deleted). Update it to find rules relative to the repo root, or from the plugin cache, or from the installed package data.

Since the package is run via `uv run --with git+...`, the rules need to be accessible. The simplest approach: include `rules/` in the wheel via `pyproject.toml` build config.

- [ ] **Step 1: Update pyproject.toml to include rules/ in the wheel**

Add to `[tool.hatch.build.targets.wheel]`:
```toml
[tool.hatch.build.targets.wheel]
packages = ["src/taleemabad_data_mcp"]

[tool.hatch.build.targets.wheel.shared-data]
"rules" = "taleemabad_data_mcp/rules"
```

Actually, the simpler approach: move rules back into the package but as a symlink, OR use `importlib.resources`. The cleanest: keep rules in `rules/` at repo root, add it to the wheel as package data.

Alternative (simpler): **Put the canonical rules in `src/taleemabad_data_mcp/rules/`** (keep the src copy as source of truth), and sync TO `rules/` at repo root for plugin visibility. This is actually what the current bump command does вЂ” it syncs src в†’ rules/. This means `src/` IS the source of truth and `rules/` is the derived copy.

**Decision: Keep `src/taleemabad_data_mcp/rules/` as source of truth.** The repo-root `rules/` is the derived copy for plugin agents to read. The bump command already syncs this. `.claude/rules/` is the dev copy.

- [ ] **Step 1 (revised): Do NOT remove src rules. Instead, remove .claude/rules/ from git tracking**

The `.claude/rules/` directory should not be tracked in git. It's a dev convenience copy that should be generated, not committed.

```bash
# Check if .claude/rules is tracked
git ls-files .claude/rules/ | head -5
```

If tracked, remove from git but keep locally:
```bash
git rm -r --cached .claude/rules/
echo ".claude/rules/" >> .gitignore
```

- [ ] **Step 2: Update bump command to also sync to .claude/rules/ for dev**

In `cli.py:bump_version()`, after syncing to `rules/`, also sync to `.claude/rules/`:

```python
# Sync to .claude/rules/ for dev convenience
claude_rules_dir = repo_root / ".claude" / "rules"
if claude_rules_dir.parent.exists():
    if claude_rules_dir.exists():
        shutil.rmtree(claude_rules_dir)
    shutil.copytree(src_rules_dir, claude_rules_dir)
```
