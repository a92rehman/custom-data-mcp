The Big Picture
What happens when you ask a question?
Every data question follows this 6-step journey. Each step has guardrails — and if anything fails, the self-healing loops kick in.
1

You Ask a Question User

Type your question in Claude Code in plain English. Examples:

"How many active PRIMARY teachers are in ICT?"
"Show me LP adoption rate for session 2025-26"
"What's the average FICO Section B score this month?"
2

data-analyst Reads Rules Phase 1

The data-analyst agent wakes up and reads the governance rule files. It finds the right rules based on your region (ICT, Rawalpindi, Moawin/Akhuwat, MySchool) and domain (teachers, lesson plans, coaching, training, etc.).

Reads ~/.claude/taleemabad-rules-path to find rules → opens index.md → reads domain-specific rule file (e.g. ict-islamabad/dimensions/teachers/teacher-query-rules.md)
3

Asks You Clarification Questions Phase 2

The rule file defines mandatory questions that must be answered before generating SQL. The agent asks you — never assumes.

Common questions:
• "Which teacher level? PRIMARY, MIDDLE, SECONDARY, or all?"
• "Which region? ICT/Islamabad or another?"
• "Which academic session? 2024-25 or 2025-26?"
• "What time period? (never assumes a duration)"
4

Generates Governed SQL Phase 3

After you answer, the agent generates SQL following the rule file exactly — correct tables, joins, filters, partition keys. Never ad-hoc SQL.

Includes: partition filter (always), is_active filters, is_testing_account = "false", correct canonical table (analytics_events not the banned analytics_analyticsevent), proper aggregation pattern from the rule file.
5

MCP Server Executes on BigQuery Railway

The parent session sends the SQL to the MCP server (running on Railway) via execute_query. The server enforces cost guardrails, runs the query, and logs everything to the audit trail.

What the server checks:
• Cost estimate — rejects if above maximum_bytes_billed
• Banned tables — blocks analytics_analyticsevent (68 GB unpartitioned)
• Auth — validates email domain in remote mode
• Audit — logs query, cost, tables accessed, user, timestamp

Returns structured JSON:
Success: {"status":"ok", "rows":[...], "event_id":"...", "cost_usd":0.001}
Failure: {"status":"error", "error_class":"SCHEMA_DRIFT", "message":"..."}
6

You Get Your Answer Done

Results are presented to you with data freshness timestamp, cost, and the rule file used. Every query is traceable from question to result.

Every response includes: data freshness ("as of 2 hours ago"), cost ($0.003), domain classification, audit event ID. You can submit feedback (thumbs up/down) or export to CSV/JSON.
When Things Break
Two self-healing loops catch and fix problems
Query errors get auto-fixed. Infrastructure issues get auto-diagnosed. Everything is tracked as tickets.

Loop 1: Query Self-Healing

When a SQL query fails, the system tries to fix it automatically

1
Query fails — execute_query returns structured error JSON with error_class
2
Opens a ticket — records the error, SQL hash, and event ID via report_ticket()
3
Dispatches query-fixer — sends: original question, failed SQL, error JSON, table schema, rule file path
4
query-fixer diagnoses — reads actual schema, compares to rule file, proposes corrected SQL (SQL v2)
5
Re-executes SQL v2 — if it works, ticket closed as auto_fixed. If not, retry up to 3 times total.
6
After 3 failures — ticket escalated, user told: "Run /taleemabad-doctor or dispatch data-admin"

Loop 2: System Self-Healing

When infrastructure breaks, system-doctor detects and fixes it

1
Hook runs on session start — update.py syncs rules, checks health (env, paths, versions)
2
Issues detected? — writes symptom IDs to sentinel file: ~/.claude/taleemabad-doctor-needed
3
system-doctor activates — auto-triggered by sentinel, or manually via /taleemabad-doctor
4
DETECT → DIAGNOSE → HEAL — matches each symptom to its handler, applies the fix, verifies it worked
5
Verification passes? — ticket closed as auto_fixed. Each fix attempt is logged.
6
After 2 failed fixes — ticket escalated to GitHub issue (auto-filed with sanitized evidence)
The Team
4 Agents, each with a clear job
Each agent is a separate subprocess with specific tools and responsibilities. They work together but never step on each other's toes.
📊

data-analyst

The primary agent — handles all data questions
When it's used
  • Any question about Taleemabad data
  • Teacher counts, LP rates, FICO scores, training progress
  • Data from ICT, Rawalpindi, Moawin/Akhuwat, MySchool
  • "What data do you have?" / "What regions?"
What it does
  • Phase 1: Finds and reads governance rules
  • Phase 2: Asks mandatory clarification questions
  • Phase 3: Generates governed SQL
  • Phase 4: Retries on error (dispatches query-fixer)
ReadGlobGrep
🛠

data-admin

The detective — investigates infrastructure
When it's used
  • Table schemas: "What columns does X have?"
  • Freshness: "When was coaching_observation updated?"
  • Audit logs: "Show me yesterday's query costs"
  • Troubleshooting: "Why is my query failing?"
What it does
  • Browses schemas with get_table_schema
  • Checks freshness with check_table_freshness
  • Queries audit logs for cost and error patterns
  • Compares actual schema to rule file definitions
All MCP tools
🔧

query-fixer

The mechanic — fixes broken SQL queries
When it's used
  • Auto-dispatched when execute_query returns an error
  • Only for query-level errors (not system issues)
  • Never called directly by users
What it does
  • SCHEMA_DRIFT: Finds renamed columns from actual schema
  • MISSING_PARTITION: Adds required date filter from rule file
  • SYNTAX_ERROR: Fixes typos, backticks, function names
  • COST_EXCEEDED: Narrows date range or switches table
  • Cannot execute queries — only proposes fixes
  • Gives up after 3 attempts (never returns same SQL)
ReadGlobGrep
⚡

system-doctor

The healer — fixes infrastructure issues
When it's used
  • Auto-triggered by sentinel file from session hook
  • Manually via /taleemabad-doctor command
  • When query errors are system-level (TIMEOUT, UNAVAILABLE)
What it does
  • Checks MCP server connectivity (curl /health)
  • Verifies env file and TALEEMABAD_USER variable
  • Ensures governance rules are synced and up-to-date
  • Detects and fixes session hook crashes
  • Checks plugin installation and version
  • Escalates to GitHub after 2 failed fix attempts

System Doctor
9 symptoms it knows how to handle
Each symptom has a detection method, an auto-fix strategy, and a fallback if auto-fix fails.
SymptomHow It's DetectedAuto-FixIf Auto-Fix Fails
connection_failed Curl the MCP server health endpoint — anything other than HTTP 200 with {"status":"ok"} Wait 10 seconds, retry once Tell user to check internet connection
user_env_missing Check if ~/.claude/taleemabad-data-mcp.env exists and contains TALEEMABAD_USER Read the most recent entry from audit log, extract email, rewrite env file Ask user for email → run /taleemabad-setup
user_env_unexpanded MCP headers contain the literal string ${TALEEMABAD_USER} instead of an email Run setx (Windows) or append to .bashrc/.zshrc (Linux/Mac) Tell user to restart their terminal (setx only affects new processes)
rules_path_missing ~/.claude/taleemabad-rules-path doesn't exist, or points to a directory without index.md Re-run the Python session hook (update.py) to repopulate Suggest full plugin reinstall after 2 failures
rules_stale Compare local version in ~/.claude/taleemabad-rules-version with latest git tag — alert if 2+ versions behind Delete version file (forces refresh), then re-run session hook —
plugin_not_installed Run claude plugin list and check if taleemabad-data appears — Print the two install commands for the user to run
plugin_outdated Local plugin version is 2+ minor versions behind the latest git tag — Suggest claude plugin update taleemabad-data@Orenda-Project
mcp_handshake_fail Server is up (health check passes) but the MCP protocol connection fails Retry once after 5 seconds Escalate to GitHub — may be auth or protocol mismatch
hook_crashed bash.exe.stackdump found in repo, or ~/.claude/taleemabad-hook.log shows errors Verify Python hook (update.py) is preferred over bash, delete stackdump —
Error Classification
8 error classes drive the fix strategy
When execute_query fails, the error is classified and routed to the right agent.
SCHEMA_DRIFT
Column renamed or removed, table not found
→ query-fixer reads schema, finds replacement
MISSING_PARTITION
Query on partitioned table without date filter
→ query-fixer adds filter from rule file
SYNTAX_ERROR
SQL typo, wrong function, mismatched parens
→ query-fixer corrects syntax
COST_EXCEEDED
Query scans more bytes than allowed limit
→ query-fixer narrows date range or switches table
PERMISSION_DENIED
No access to the table or dataset
→ system-doctor (not a query problem)
TIMEOUT
Query deadline exceeded
→ system-doctor (infrastructure issue)
BIGQUERY_UNAVAILABLE
503 / BigQuery service down
→ system-doctor (service outage)
OTHER
Unrecognized exception type
→ Manual diagnosis needed
